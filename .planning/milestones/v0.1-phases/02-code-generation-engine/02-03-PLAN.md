Create `CodeGeneratorTest.kt` in `codegen/src/test/kotlin/com/example/declarativejooq/codegen/`:

The test class uses kotlin-compile-testing to compile generated source and execute it against an H2 database. The key challenge is that generated code must be invoked via reflection since it's compiled by a separate classloader.

```kotlin
package com.example.declarativejooq.codegen

import com.tschuchortdev.compiletesting.KotlinCompilation
import com.tschuchortdev.compiletesting.SourceFile
import org.junit.jupiter.api.Test
import org.junit.jupiter.api.BeforeEach
import org.junit.jupiter.api.Assertions.*
import java.io.File

class CodeGeneratorTest {
    private val classDir = File("../dsl-runtime/build/classes/kotlin/test")
    private val outputPackage = "com.example.generated"
    private lateinit var generatedSources: List<Pair<String, String>>

    @BeforeEach
    fun setUp() {
        assertTrue(classDir.exists(), "Run ./gradlew :dsl-runtime:testClasses first")
        generatedSources = CodeGenerator().generateSource(classDir, outputPackage, "com.example.declarativejooq")
        assertTrue(generatedSources.isNotEmpty(), "CodeGenerator produced no source files")
    }
    // ... tests below
}
```

**Test 1: `generatedCodeCompiles`**
- Creates `KotlinCompilation()` with `sources = generatedSources.map { (name, code) -> SourceFile.kotlin(name, code) }`, `inheritClassPath = true`, `messageOutputStream = System.out`
- Calls `.compile()`
- Asserts `result.exitCode == KotlinCompilation.ExitCode.OK` — if not, print `result.messages` for debugging
- This covers CODEGEN-03, CODEGEN-04, CODEGEN-05 at the compilation level

**Test 2: `generatedBuilderHasTypedProperties`**
- Compile the generated sources (same as test 1)
- Load the organization builder class: `result.classLoader.loadClass("$outputPackage.OrganizationBuilder")`
- Assert the class has declared fields/methods for the `name` property (Kotlin `var` compiles to getter+setter)
- Load the app_user builder class: `result.classLoader.loadClass("$outputPackage.AppUserBuilder")`
- Assert it has `name` and `email` properties
- This covers CODEGEN-03 (typed property setters)

**Test 3: `generatedCodeExecutesAgainstDatabase`**
- Compile the generated sources
- Use `com.example.declarativejooq.TestSchema.createDslContext()` to get an H2 DSLContext (TestSchema is on the test classpath via `testImplementation(project(":dsl-runtime"))`)
- Execute the generated DSL via reflection:
  - Load the file class containing the DslScope extension: `result.classLoader.loadClass("$outputPackage.OrganizationBuildersKt")` (KotlinPoet generates a file class named after the FileSpec)
  - Find the extension function method: look for a static method named "organization" that takes DslScope as first param
  - Actually, a simpler approach: use the `execute` function from dsl-runtime directly. The generated extension functions are on DslScope — but since they're loaded by a different classloader, we need to be careful.
  - **Practical approach:** Instead of reflection gymnastics, generate a small test-only Kotlin source that imports both dsl-runtime and the generated builders, runs `execute(ctx) { organization { name = "Test" } }`, and returns the result. Compile THIS source alongside the generated sources. Then invoke the test function via reflection.
  - Create an additional `SourceFile.kotlin("TestHarness.kt", ...)` containing:
    ```kotlin
    package com.example.generated
    import com.example.declarativejooq.*
    object TestHarness {
        fun run(ctx: org.jooq.DSLContext): DslResult {
            return execute(ctx) {
                organization {
                    name = "Acme"
                    appUser {
                        name = "Alice"
                        email = "alice@acme.com"
                    }
                }
            }
        }
    }
    ```
    NOTE: The function names (`organization`, `appUser`) and property names come from the generated code. The test must use the actual generated function names. Since we generate from TestSchema which has `organization` and `app_user` tables, the generated names will be `organization` (root extension) and `appUser` (child builder function inside OrganizationBuilder).
  - Compile generated sources + TestHarness together
  - Load `com.example.generated.TestHarness` from result.classLoader
  - Invoke `run(ctx)` via reflection, passing the H2 DSLContext
  - The returned `DslResult` can be examined directly since `DslResult` class is from the parent classloader
  - Assert the result contains 1 organization record and 1 app_user record
  - Query the H2 DB directly to verify: `ctx.selectFrom("organization").fetch()` has 1 row, `ctx.selectFrom("app_user").fetch()` has 1 row

**Test 4: `fkResolutionWorks`**
- Same setup as test 3 (compile + TestHarness + execute)
- After execution, query the DB:
  - `val org = ctx.selectFrom("organization").fetchOne()` — get the org ID
  - `val user = ctx.selectFrom("app_user").fetchOne()` — get the user's organization_id
  - Assert `user.get("organization_id") == org.get("id")` — FK was resolved correctly
- This covers CODEGEN-06 (nested FK builder resolution)

**Test 5: `generatedDslResultHasTypedAccessors`**
- Same compile + execute as tests 3-4
- Generate a second TestHarness that also creates a `GeneratedDslResult` wrapper:
    ```kotlin
    object TestHarness {
        fun run(ctx: org.jooq.DSLContext): Any {
            val result = execute(ctx) { organization { name = "Acme" } }
            return GeneratedDslResult(result)
        }
    }
    ```
  - Or include both in one harness: `run` returns DslResult, and `runWithTypedResult` returns GeneratedDslResult
- Load the result, invoke the typed accessor via reflection: call `organizations()` method
- Assert it returns a non-empty list
- Get first element, call `name` getter via reflection, assert it equals "Acme"
- This covers CODEGEN-04 (result class with typed accessors) and CODEGEN-05 (DslResult typed wrapper)

**Important implementation notes:**
- Use `@BeforeEach` to clean the H2 database between tests: each test that touches DB should use a fresh `TestSchema.createDslContext()` or truncate tables
- Handle the case where each test creates its own H2 database instance (since TestSchema uses `DB_CLOSE_DELAY=-1`, repeated calls reuse the same DB — add `DROP TABLE IF EXISTS` before `CREATE TABLE IF NOT EXISTS`, or use unique DB names per test)
- If kotlin-compile-testing has issues with Kotlin 2.x, the test will fail at compilation — this is the early detection point mentioned in the research. If it fails, add a comment noting the ZacSweers fork as fallback.
- The TestHarness approach avoids complex reflection across classloader boundaries. The generated code and the test harness are in the SAME classloader (the KotlinCompilation result classloader), so they can import each other naturally. Only the invocation from the test method uses reflection to cross the boundary.
