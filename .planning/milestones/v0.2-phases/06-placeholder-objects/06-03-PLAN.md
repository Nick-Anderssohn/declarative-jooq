IMPORTANT: The harness above uses root-level builders for cross-tree scenarios. The `createdBy { }` function would only exist as a child builder on AppUserBuilder, not on DslScope. We need to rethink the cross-tree harness.

Cross-tree means records from separate root trees. The actual syntax would be:
```kotlin
execute(ctx) {
    val org = organization { name = "Acme" }
    val alice = organization {
        name = "Acme-host"
        appUser {
            name = "Alice"
            email = "alice@acme.com"
        }
    }
    // Wait — appUser returns AppUserResult, but it's nested inside organization
    // We need: capture alice from inside the nested block
}
```

Actually, the natural cross-tree test is:
```kotlin
execute(ctx) {
    val org = organization { name = "Acme" }
    // Tree 1: org with Alice
    organization {
        name = "Host"
        val alice = appUser {
            name = "Alice"
            email = "alice@acme.com"
        }
        // Can use alice here within same tree — that's same-tree
    }
    // Tree 2: another org with task referencing Alice — but task isn't a root builder
}
```

The cross-tree scenario from CONTEXT.md uses `val alice = appUser { }` at root level then references it. But `appUser` has a required FK `organization_id`, so it needs an org. The placeholder approach handles this:

```kotlin
execute(ctx) {
    val org = organization { name = "Acme" }
    val alice = appUser {
        name = "Alice"
        email = "alice@acme.com"
        organization = org  // placeholder FK wiring
    }
}
```

Wait, is `appUser` a root-level DslScope function? Looking at TableIR: `isRoot = outboundFKs.none { !it.isSelfReferential }`. `app_user` has outbound FK `organization_id` which is not self-referential, so `isRoot = false`. Therefore `appUser` is NOT a DslScope extension function — it only exists as a child builder function inside OrganizationBuilder.

So cross-tree for tasks would be:
```kotlin
execute(ctx) {
    val org = organization {
        name = "Acme"
        val alice = appUser {
            name = "Alice"
            email = "alice@acme.com"
            // organization_id auto-resolved from parent org
        }
        appUser {
            name = "Bob"
            email = "bob@acme.com"
            createdBy {
                title = "Bob's task created by Alice"
                createdBy = alice  // alice is from a sibling subtree, technically same root tree
            }
        }
    }
}
```

For TRUE cross-tree (two root trees):
```kotlin
execute(ctx) {
    val alice: AppUserResult
    organization {
        name = "Acme"
        alice = appUser {
            name = "Alice"
            email = "alice@acme.com"
        }
    }
    organization {
        name = "Beta"
        appUser {
            name = "Bob"
            email = "bob@beta.com"
            createdBy {
                title = "Cross-tree task"
                createdBy = alice  // from Acme tree, used in Beta tree
            }
        }
    }
}
```

But `alice` is a `val` declared inside the first `organization` block. To use it outside, we'd need to capture it before the block. Since child builder functions now return Result, we can capture from the return:

```kotlin
val aliceRef: AppUserResult
organization {
    name = "Acme"
    aliceRef = appUser {
        name = "Alice"
        email = "alice@acme.com"
    }
}
```

But Kotlin requires `aliceRef` to be assigned before use. Using `lateinit var`:
```kotlin
lateinit var alice: AppUserResult
organization {
    name = "Acme"
    alice = appUser { ... }
}
organization {
    name = "Beta"
    appUser {
        name = "Bob"
        email = "bob@beta.com"
        createdBy {
            title = "Cross-tree task"
            createdBy = alice
        }
    }
}
```

This works because the block runs immediately (eagerly). By the time we reach the second `organization` block, `alice` is assigned.

Wait, the `override` test has a problem. The `organization = beta` placeholder property is set on the AppUserBuilder. But `organization_id` is also being auto-resolved from the parent context (Acme-again). The placeholder override happens AFTER parent FK resolution in TopologicalInserter. So the test should work: placeholder wins.

But there's a subtlety: `organization { name = "Acme-again" }` creates a root tree. The `appUser` nested inside has `parentNode = Acme-again's org node`. The parent FK resolution sets `organization_id = Acme-again.id`. Then placeholder resolution sees the `organization` placeholder set to `beta` and overrides `organization_id = beta.id`. This is PLCH-04. Good.

However, `val acme = organization { name = "Acme" }` and `val beta = organization { name = "Beta" }` are root-level calls that return `OrganizationResult`. These are root builders. The DslScopeEmitter generates `fun DslScope.organization(block: ...) : OrganizationResult`. So `val acme` and `val beta` capture works. The third `organization { }` doesn't capture its return.

**Capturing placeholders:**
```kotlin
val result = execute(ctx) {
    organization {
        name = "Acme"
        val alice = appUser {
            name = "Alice"
            email = "alice@acme.com"
        }
        // alice is an AppUserResult — alice.name returns "Alice" immediately
        // alice.id returns null until execute() completes
    }
}
// After execute(): alice.id is populated with the DB-generated value
```

Brief explanation: Every builder block returns a typed Result object. Capture it with `val` when you need to reference it elsewhere. Ignore the return value when you don't (Kotlin discards it silently).

**Cross-tree FK wiring:**
```kotlin
val result = execute(ctx) {
    lateinit var alice: AppUserResult
    organization {
        name = "Acme"
        alice = appUser {
            name = "Alice"
            email = "alice@acme.com"
        }
    }
    organization {
        name = "Beta"
        appUser {
            name = "Bob"
            email = "bob@beta.com"
            createdBy {
                title = "Cross-org task"
                createdBy = alice  // Alice from Acme tree, task from Beta tree
            }
        }
    }
}
```

Brief explanation: Placeholders work across root trees within the same `execute` block. The library automatically adjusts insert order so the referenced record is inserted first.

**Overriding parent-context FK:**
```kotlin
val result = execute(ctx) {
    val targetOrg = organization { name = "Target" }
    organization {
        name = "Host"
        appUser {
            name = "Bob"
            email = "bob@target.com"
            organization = targetOrg  // overrides auto-resolved FK from Host
        }
    }
}
// Bob's organization_id = Target.id, not Host.id
```

Brief explanation: When a builder is nested under a parent, the FK is auto-resolved to the parent. Setting a placeholder property on the same FK field overrides the auto-resolved value.

**Fan-out (one-to-many references):**
```kotlin
val result = execute(ctx) {
    organization {
        name = "Acme"
        val alice = appUser {
            name = "Alice"
            email = "alice@acme.com"
        }
        appUser {
            name = "Worker"
            email = "worker@acme.com"
            createdBy {
                title = "Task 1"
                createdBy = alice
            }
            createdBy {
                title = "Task 2"
                createdBy = alice
            }
        }
    }
}
// Both tasks have created_by = Alice's id
```

Brief explanation: A single placeholder can be assigned to FK properties on multiple builders. All referencing records will point to the same placeholder's PK.
